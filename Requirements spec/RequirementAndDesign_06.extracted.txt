CHAPTER I. FUNCTIONAL REQUIREMENTS
The Student Management System provides core features categorized by user roles and permissions (Admin, Student, Lecturer).
Authentication & Authorization
Registration: The system does not allow students or lecturers to register their own accounts. Only the Administrator has the right to create accounts based on admission or personnel records. Mandatory fields include: National ID (CCCD), Full Name, Date of Birth, Email, Gender, Phone Number, and Faculty/Major.
Username & Password Initialization: The Username will be automatically generated by the system based on the user's Student ID or Employee ID. The initial password is randomly generated and automatically sent to the student's or lecturer's email/phone number.
Log in: Allows all users to log in using their Username and Password. Upon successful authentication, the system automatically identifies the role (Admin, Student, Lecturer) to display the corresponding interface and features. If the credentials are incorrect, the system will display an error message.
Forgot Password: Users can request a password reset via their registered Email or Phone number. The system will send a secure, time-limited link or OTP to verify identity before allowing a new password to be set.
Log out: Allows users to log out at any time. The system will terminate the session and clear temporary data to ensure security.
Change Password: Users are required to enter their current password before setting a new one. The new password must meet complexity requirements (e.g., minimum 8 characters, including uppercase, numbers, and special characters). The system will require confirmation before saving the changes.
User Management
Student Management: The Admin has the right to view, add, update, and delete student records (CRUD). Each student must have a Status. Students can view and update their own personal contact information (Email, Phone Number), but are restricted from modifying identification data such as Student ID, National ID, and affiliated Faculty.
Lecturer Management: Similar to students, the Admin manages all records of the teaching staff. Lecturers can update basic personal information themselves, excluding their affiliated Faculty. The Admin is responsible for assigning lecturers to specific class sections.
Academic Management
Faculty Management: The Admin has the right to manage the Faculty catalog, including Faculty Code, Faculty Name, Email, and Phone Number.
Semester/Term Management: The Admin manages academic semesters including start dates, end dates, and event timelines (registration periods, exam weeks). All academic operations are tied to a specific semester.
Course Management: The Admin is allowed to view, create, edit, and delete courses. Course data must include: Course Code, Course Name, Credits, Managing Faculty, and Prerequisite Constraints.
Class/Section Management: A course can be opened as multiple class sections within a specific Semester. The Admin manages classes with: Class Code, Course Code, Semester, Maximum Capacity, Current Enrollment, and a specific Schedule (Day of the week, Start Time to End Time).
View Training Program (Curriculum): Students can view the standard curriculum framework for their major. The system displays the list of courses, credits, and prerequisites, and automatically labels the status as "Completed" or "Incomplete" based on the student's transcript.
Course Registration
Registration Period Control: The Admin has the right to open and close the course registration portal for a specific Semester.
Automated Course Filtering: When the portal is open, students view the list of available classes for the selected semester. The system automatically hides courses that the student has already completed.
Registration Eligibility Validation Algorithm: When a student attempts to register, the system must automatically check 4 constraints:Prerequisites: Has the student passed the prerequisite course?
Credit Limit: Does the total number of credits exceed the configured maximum limit for the semester?
Capacity: Has the class reached its maximum enrollment limit?
Schedule Conflict: Does the schedule of the selected class conflict (same day, overlapping time) with any previously registered classes?
Process Registration/Cancellation: If all conditions are met, the system records the registration and increases the class enrollment count by 1. If any constraint is violated, the system rejects the request and displays a detailed error message explaining the reason. Students can cancel a registered class within the permitted timeframe.
Grading & Attendance Management
Attendance: The system allows lecturers to take attendance for students in each session, track absences, and automatically calculate the attendance rate.
Gradebook: Lecturers can enter component grades (midterm, assignments, attendance) and final exam grades. The system automatically calculates the final summary grade based on the course's weighted formula, and assigns the letter grade (A, B, C, D, F).
Grade Finalization: Lecturers can save grades as "Draft". Once the grading deadline ends, the Admin or authorized role executes the "Finalize Grades" function. After finalization, grades cannot be altered.
GPA Calculation: The system automatically calculates both Semester GPA and Cumulative GPA based on the finalized grades.
CHAPTER II. NON-FUNCTIONAL REQUIREMENTS
Security: User passwords must be hashed before being stored in the database. Sessions must automatically expire after a period of inactivity. The system must strictly comply with Role-Based Access Control (RBAC).
Usability: The system interface must be intuitive and responsive. Error messages must be clear and guide users on how to fix the issue (e.g., highlighting the specific field causing the error or specifying which course causes a schedule conflict).
Reliability: Ensure data integrity and consistency through Transaction management (ACID). Data loss is strictly prohibited, even in the event of a system restart due to a crash. The system requires regular automated database backups.
Performance: The system must be capable of handling high concurrent access during course registration peaks without crashing. Response time for any user action must not exceed 2 seconds under normal load.
Maintainability: The system needs to be developed using a modular architecture to easily facilitate future upgrades and feature expansions. Code must be documented and follow standard naming conventions.
Design Constraints: The Student Management System using programming language Python.

DFD Level 2: Further decomposes Process 4.0 (Course Registration) into 4 sub-processes, detailing the step-by-step logic of the registration workflow.

CHAPTER VII. INTERFACE DESIGN DESCRIPTION
Overall Interface Overview
The Student Management System (SMS) is implemented as a modern application with a graphical user interface (GUI). The system interface must be intuitive and responsive. Users interact with the system through visual forms, dashboards, and navigation menus rather than text-based console inputs. Based on user roles, including the admin, the student, and the lecturer, the system provides separate menus with functions appropriate to each role. The system must strictly comply with Role-Based Access Control (RBAC).
Main Menu & Authentication Design
When the application starts, the system displays a main authentication menu that serves as the entry point for all users. The main menu provides the following options
Log in using a Username and Password.
Forgot Password to request a password reset via registered Email or Phone number. The system will send a secure, time-limited link or OTP to verify identity.
Upon successful authentication, the system automatically identifies the role (Admin, Student, Lecturer) to display the corresponding interface and features.
Role-Based Menu Design
To ensure clear separation of responsibilities and secure access control, the system implements role-based menus. Each role is provided with a distinct set of functions.
3.1 Administrator Interface
The administrator interface allows administrators to manage academic data, system configurations, and user accounts. Main functions provided to administrators include
Creating accounts based on admission or personnel records.
Managing student and lecturer profiles (view, add, update, delete).
Assigning lecturers to specific class sections.
Managing the Faculty catalog, Semesters/Terms, Courses, and Class/Sections.
Controlling the registration period by opening and closing the portal.
Executing the "Finalize Grades" function once the grading deadline ends.
Changing passwords and logging out of the system.
3.2 Student Interface
The student interface is designed to support course registration and academic planning activities. Students can perform the following actions
Viewing and updating their own personal contact information (Email, Phone Number).
Viewing the standard curriculum framework for their major, with statuses automatically labeled as "Completed" or "Incomplete".
Viewing the list of available classes for the selected semester.
Registering for courses and canceling registered classes within the permitted timeframe.
Changing passwords and logging out of the system.
3.3 Lecturer Interface
The lecturer interface provides basic teaching-related functionalities. Lecturers are able to
Updating basic personal information, excluding their affiliated Faculty.
Taking attendance for students in each session and tracking absences.
Entering component grades and final exam grades and saving grades as "Draft"
Changing passwords and logging out of the system.
User Interaction and System Feedback
The system provides immediate and clear feedback after each user action.
If an operation is completed successfully, the system proceeds with the action, such as requiring confirmation before saving password changes or displaying a success message.
If an operation fails or credentials are incorrect, the system will display an error message.
Error messages must be clear and guide users on how to fix the issue.
The system provides specific visual feedback, such as highlighting the specific field causing the error.
During course registration, the system provides detailed error messages explaining the reason if constraints (Prerequisites, Credit Limit, Capacity, Schedule Conflict) are violated. For example, the system will specify which course causes a schedule conflict.
Interface Design Rationale
A responsive Graphical User Interface (GUI) was selected for this system due to the following reasons
The non-functional requirements explicitly mandate that the system interface must be intuitive and responsive.
The system requires advanced visual error handling, such as highlighting specific fields causing errors, which is effectively implemented in GUI or web-based environments rather than standard console applications.
The interface effectively manages the display of complex academic data, such as curriculum frameworks, automated course filtering, and gradebooks.

4.1 | Registration Period Control | Controls the open/close status of the registration portal per semester. Admin activates or deactivates the portal; this process updates the status in D4 (Semester) and returns a confirmation to Admin.
4.2 | Automated Course Filtering | Automatically filters the list of class sections displayed to students: hides courses already completed, and shows only sections belonging to the currently open registration semester. Reads from D6, D8 (completed grades), and D4 (semester status).
4.3 | Registration Eligibility Validation | Validates 4 registration constraints in sequence: (1) all prerequisite courses completed, (2) no schedule conflict with existing registrations, (3) total registered credits do not exceed the maximum, (4) the class section still has available seats. All 4 must pass before proceeding to Process 4.4.
4.4 | Process Registration / Cancellation | Executes the actual enrollment or cancellation: creates or removes a record in D7 (Registration), updates the current enrollment count in D6 (Class/Section), and returns the outcome to the student.
